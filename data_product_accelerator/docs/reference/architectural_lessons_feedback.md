## 1. Context

The [skill_feedback_report.md](skill_feedback_report.md) documented 18 runtime errors from the initial optimization session -- errors like `FileNotFoundError`, `MlflowException`, API sorting failures, and premature lever application. Those were point fixes.

This document covers a **second wave of architectural insights** that emerged after the runtime errors were resolved. These are not individual bugs but systemic design gaps that required coordinated multi-file, cross-skill changes. They represent deeper structural issues in how the 5 skills (Orchestrator, Evaluator, Optimizer, Applier, Generator) communicate data.

The common theme: **data was generated but never arrived at its consumer.** Structured ASI was emitted by judges but lost in serialization. Traces were logged but not queryable. Proposals were generated but never converted to executable patches. The pipeline had all the right components -- but the wiring between them was broken.

## Lesson 1: ASI Pipeline Broken by Serialization Gap

**Source:** [ASI Pipeline Tightening Plan](../../plans/asi_pipeline_tightening_2b384190.plan.md)

### The Problem

Judges produced structured Actionable Side Information (ASI) in `Feedback.metadata`, but the optimizer read from `eval_results` rows which did NOT contain the `assessments` list. The data was generated by the evaluator but never arrived at the optimizer. Additionally, 3 LLM judges (`schema_accuracy`, `logical_accuracy`, `semantic_equivalence`) and the arbiter produced no structured ASI at all -- just free-text rationale.

### Root Cause

No end-to-end contract existed between evaluator output and optimizer input. Each skill described its own I/O independently:
- The evaluator SKILL.md described `Feedback.metadata` as the ASI carrier
- The optimizer SKILL.md described reading `metadata/{judge}/failure_type` columns
- Neither documented the serialization boundary: how `mlflow.genai.evaluate()` surfaces `Feedback.metadata` in result DataFrames

The actual behavior: `Feedback.metadata` appears as `{judge}/metadata` columns in `eval_result.tables["eval_results"]`, but the `assessments` list (which contains the full metadata object) is NOT reliably available in the DataFrame. The optimizer was reading column names that didn't exist.

### What Was Fixed

1. Created UC Delta table `genie_eval_asi_results` as the canonical ASI storage
2. Evaluator writes one row per (question, judge) pair with full ASI metadata after every evaluation
3. Optimizer reads from UC table as primary source via `read_asi_from_uc()`
4. Fallback chain: UC table -> `{judge}/metadata` columns -> assessments list -> regex on rationale

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Orchestrator | Hard constraint: "Evaluator MUST persist ASI to UC table. Optimizer MUST read from UC table as primary source." |
| Evaluator | Document the serialization boundary and UC table write contract |
| Optimizer | Document the priority chain for ASI consumption |

### Generalized Principle

> **Cross-skill data contracts must specify the transport mechanism, not just the schema.** Knowing that judge X produces `failure_type` is insufficient -- the contract must also specify WHERE the consumer reads it (UC table, artifact, column name) and WHAT happens at serialization boundaries.

## Lesson 2: MLflow Traces Should Be Stored in Unity Catalog

**Source:** [Databricks docs: trace-unity-catalog](https://docs.databricks.com/aws/en/mlflow3/genai/tracing/trace-unity-catalog)

### The Problem

MLflow traces were stored only in the default MLflow artifact store (DBFS or managed), which is:
- Opaque to SQL queries -- cannot `SELECT * FROM traces WHERE judge='arbiter'`
- Not governed by Unity Catalog access controls
- Not available for cross-experiment analysis or dashboards
- Difficult to correlate with the `genie_eval_asi_results` UC table

### What Was Implemented

In `run_genie_evaluation.py` (lines 314-346):

```python
from mlflow.entities import UCSchemaLocation
from mlflow.tracing.enablement import set_experiment_trace_location

_uc_location = UCSchemaLocation(
    catalog_name=_uc_trace_catalog,
    schema_name=_uc_trace_schema,
)
set_experiment_trace_location(
    location=_uc_location,
    experiment_id=exp.experiment_id,
)
mlflow.tracing.set_destination(destination=_uc_location)
set_databricks_monitoring_sql_warehouse_id(
    warehouse_id=warehouse_id,
    experiment_id=exp.experiment_id,
)
```

This requires `mlflow[databricks]>=3.9.0`.

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Evaluator | Add "UC Trace Storage" section with 3-step setup and version requirement |
| Orchestrator | Hard constraint: "Evaluator MUST configure UC trace storage before evaluation" |
| Job YAML | Document `mlflow[databricks]>=3.9.0` dependency |

### Generalized Principle

> **Observability data should live in the same governance layer as the data it observes.** MLflow traces about Genie Space SQL generation should be queryable alongside the UC tables that Genie reads, enabling end-to-end lineage.

## Lesson 3: Applier Had Two Disconnected Execution Paths

**Source:** User observation during conversation

### The Problem

The applier had two completely disconnected code paths:
1. **Patch DSL path** (`use_patch_dsl=True`): `render_patch()` -> `classify_risk()` -> `apply_patch_set()` with risk gating and rollback
2. **Legacy path** (`use_patch_dsl=False`): `apply_proposal_batch()` which returned `"pending_agent_execution"` -- it did not actually apply anything

Critically, `USE_PATCH_DSL` defaulted to `False`, meaning the default path did nothing. And there was no bridge to convert the optimizer's ASI-enriched proposals into Patch DSL patches.

### What Was Fixed

1. **`USE_PATCH_DSL = True`** (was `False`) -- Patch DSL is now the default
2. **New `proposals_to_patches()` function** -- Bridges ASI-enriched proposals into Patch DSL patches using `_lever_to_patch_type` mapping:
   - `failure_type` -> patch type (e.g., `wrong_column` + lever 1 -> `update_column_description`)
   - `blame_set` -> target
   - `counterfactual_fixes` -> new_text
   - `wrong_clause`, `missing_metadata` -> optional context fields
3. **Updated `main()` CLI** -- Auto-converts proposals to patches when input has `proposals` but no `patches`
4. **Legacy path** only accessible via explicit `--no-patch-dsl`
5. **`apply_log.json` sidecar** written for rollback traceability

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Applier | Add "Patch DSL is Default" section; document `proposals_to_patches()` and `_lever_to_patch_type` mapping |
| Applier | Common Mistake: "Leaving USE_PATCH_DSL=False -> proposals return pending_agent_execution" |
| Orchestrator | Reinforce HC #18: `apply_proposal_batch()` is deprecated |

### Generalized Principle

> **Feature flags for new code paths must default to the new path once it's proven.** A `USE_PATCH_DSL = False` default meant every agent using the applier silently did nothing. The safe default should be the working path.

## Lesson 4: Optimizer Did Not Consume Structured ASI

**Source:** User's explicit request: "Can you ensure the metadata optimizer uses the ASI data output of all the judges?"

### The Problem

The optimizer had two key functions that consumed judge output, both broken:

1. **`_extract_judge_feedbacks_from_eval()`** (line 501) -- read `feedback/{judge}` columns but ignored structured ASI metadata entirely, falling back to `_infer_blame_from_rationale()` (regex on free text)
2. **`cluster_failures()`** (line 854) -- read `metadata/{judge}/failure_type` columns which didn't match the actual eval output column names (`{judge}/value`, assessments list)

The result: even when judges emitted perfect structured ASI, the optimizer never saw it. Every proposal was generated from regex-parsed rationale text.

### What Was Fixed

1. **`read_asi_from_uc()`** -- New function to query UC Delta table via Databricks SQL Statement API
2. **`_extract_judge_feedbacks_from_eval()`** -- Rewritten with UC-first priority chain
3. **`_extract_asi_from_assessments()`** -- New helper to parse assessments list
4. **`cluster_failures()`** -- Updated to use `asi_severity`, `asi_confidence`, `asi_wrong_clause`, `asi_expected_value`, `asi_actual_value`, `asi_missing_metadata`, `asi_ambiguity_detected`
5. **`generate_metadata_proposals()`** -- Each proposal now includes an `asi` dict:

```json
{
  "proposal_id": "P001",
  "lever": 3,
  "change_description": "Disambiguate routing...",
  "asi": {
    "failure_type": "repeatability_issue",
    "blame_set": "asset_routing:MV",
    "severity": "critical",
    "counterfactual_fixes": ["Strengthen NOT FOR clauses in TVF COMMENTs..."],
    "ambiguity_detected": true
  }
}
```

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Optimizer | Add "ASI Consumption Contract" section with priority chain and structured fields |
| Optimizer | Document CLI args: `--catalog`, `--schema`, `--run-id`, `--warehouse-id` |

### Generalized Principle

> **Consumer skills must be tested against actual producer output, not assumed schemas.** The optimizer's column name assumptions (`metadata/{judge}/failure_type`) never matched reality. A single integration test with real evaluator output would have caught this immediately.

## Lesson 5: Repeatability Judge Was Not Emitting Structured ASI

**Source:** User observation: "repeatability judge not following ASI output format: CRITICAL_VARIANCE: 33% consistency, 3 unique variants"

### The Problem

Cell 9c logged repeatability results as free-text rationale:
```
CRITICAL_VARIANCE: 33% consistency, 3 unique variants across 3 samples (asset=MV)
```

This text was invisible to the ASI pipeline because `_extract_judge_feedbacks_from_eval()` looked for structured metadata fields (`failure_type`, `blame_set`, `counterfactual_fix`). The repeatability information existed but was trapped in an unstructured string.

### What Was Fixed

Cell 9c now calls `mlflow.log_assessment()` with structured ASI metadata when `is_repeatable == "no"`:

```python
metadata = build_asi_metadata(
    failure_type="repeatability_issue",
    severity=severity_map[classification],  # CRITICAL -> "critical", SIGNIFICANT -> "major", MINOR -> "minor"
    confidence=1.0 - pct / 100,
    blame_set=["asset_routing:<dominant_asset>"],
    expected_value="IDENTICAL (100% consistency)",
    actual_value=f"{classification} ({pct}%, {unique_count} unique variants)",
    counterfactual_fix="Disambiguate TVF COMMENTs or add example_question_sqls...",
    ambiguity_detected=True,
)
```

Repeatability rows are also appended to the `genie_eval_asi_results` UC table.

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Evaluator | Hard constraint: "Cell 9c MUST emit structured ASI via build_asi_metadata()" |
| Evaluator | Document severity mapping from classification to ASI severity |

### Generalized Principle

> **Every data point that drives optimization must flow through the structured pipeline.** Free-text summaries are fine for human readability but must be accompanied by machine-readable structured fields for automated consumers.

## Lesson 6: LLM Judges Used make_judge Which Cannot Emit ASI

**Source:** User observation: "completeness judge not giving good structured ASI response information"

### The Problem

`make_judge()` returns an `InstructionsJudge` scorer that produces free-text `rationale` but has no mechanism to return structured `metadata` in the `Feedback` object. The 4 LLM judges that used `make_judge()` -- `schema_accuracy`, `logical_accuracy`, `semantic_equivalence`, `completeness` -- all produced unstructured output:

```
The generated SQL is missing the check_in_date dimension in the SELECT clause
and GROUP BY. The expected SQL groups by check_in_date and includes it as a
dimension, which allows revenue to be broken down by date...
```

This rationale contains useful information (missing column, wrong clause) but in a form the optimizer cannot parse reliably.

### The Fix Pattern

Convert each `make_judge()` judge to a custom `@scorer` that:

1. Constructs a prompt requesting JSON output with ASI fields
2. Calls `_call_llm_for_scoring(prompt)` via Databricks SDK
3. Parses the JSON response for `failure_type`, `wrong_clause`, `blame_set`, etc.
4. Returns `Feedback(metadata=build_asi_metadata(...))` on failure

```python
@scorer
def schema_accuracy_judge(inputs: dict, outputs: dict, expectations: dict) -> Feedback:
    prompt = f"""Analyze schema accuracy. Respond with JSON:
    {{"correct": true/false, "failure_type": "...", "wrong_clause": "...", "blame_set": [...]}}"""
    llm_result = _call_llm_for_scoring(prompt)
    if llm_result.get("correct"):
        return Feedback(name="schema_accuracy", value="yes", rationale="Schema correct")
    metadata = build_asi_metadata(
        failure_type=llm_result.get("failure_type", "wrong_column"),
        wrong_clause=llm_result.get("wrong_clause"),
        blame_set=llm_result.get("blame_set"),
        # ... other ASI fields
    )
    return Feedback(name="schema_accuracy", value="no", rationale=..., metadata=metadata)
```

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Evaluator | Add "Judge ASI Requirements" section: ALL judges MUST return structured metadata |
| Evaluator | Common Mistake: "Using make_judge() when structured ASI is needed" |

### Generalized Principle

> **Convenience APIs that hide structured output must be replaced when structure is needed downstream.** `make_judge()` is excellent for quick prototyping but cannot emit the structured metadata required by the ASI pipeline. The skill should document when to use each pattern.

## Lesson 7: Repeatability Fix Requires Bilateral Disambiguation

**Source:** [Repeatability Fix Levers Plan](../../plans/repeatability_fix_levers_4d45ff5b.plan.md)

### The Problem

Questions `rp_001` ("total revenue last quarter") and `rp_010` ("avg booking value by destination") oscillated between Metric View and TVF routing. The TVF's `COMMENT` actively claimed these question patterns:

```sql
-- get_revenue_summary TVF COMMENT (before fix):
-- BEST FOR: "What was total revenue last quarter by destination?"
-- NOT FOR: Payment-level analysis (use get_payment_analysis instead)
```

Lever 1 (UC COMMENTs on MV) and Lever 2 (MV metadata) were already applied, adding positive routing to the MV. But the TVF side was never touched -- Lever 3 was incorrectly skipped with "no proposals." The positive MV metadata competed with the TVF's unchallenged claim, causing oscillation.

### The Fix

Update `get_revenue_summary` TVF COMMENT to:
1. Remove competing claims ("total revenue last quarter" -> emphasize "by destination" breakdown)
2. Add explicit negative routing: "NOT FOR: Simple total revenue without destination breakdown (use booking_analytics_metrics MEASURE(total_revenue))"
3. Add more negative routing for avg booking value: "NOT FOR: Simple average booking value (use booking_analytics_metrics MEASURE(avg_booking_value))"

### Key Insight

When a question oscillates between two asset types, **both sides** of the disambiguation must be addressed:
- **Positive routing** (Lever 1/2): Tell the preferred asset it IS the right choice
- **Negative routing** (Lever 3): Tell the competing asset it is NOT the right choice

Applying only one side is insufficient. The optimizer must auto-detect asset-type oscillation in failure clusters and generate proposals for BOTH assets.

### Skill Updates Required

| Skill | Update |
|-------|--------|
| Orchestrator | Common Mistake: "Skipping Lever 3 when Lever 1/2 alone don't resolve oscillation" |
| Optimizer | Update "Repeatability Failure Clustering" with bilateral disambiguation pattern |

### Generalized Principle

> **Disambiguation is bilateral.** Telling asset A "you handle this" without telling asset B "you don't handle this" leaves both competing. Metadata optimization for routing must address BOTH the preferred and competing assets.

## Generalized Principles

These 7 lessons converge on 3 meta-principles:

### 1. Cross-Skill Data Contracts Must Specify Transport, Not Just Schema

Skills described WHAT data they produce (ASI fields) but not HOW it reaches the consumer (column names, UC table, artifact path). The serialization boundary between MLflow `evaluate()` and the optimizer was undocumented, causing silent data loss. **Every cross-skill contract should specify: format, location, serialization mechanism, and fallback chain.**

### 2. Default Configuration Must Be the Working Configuration

`USE_PATCH_DSL = False` meant the default code path did nothing. `make_judge()` was the default judge constructor but couldn't emit ASI. Both required explicit opt-in to the working path. **The safe default should always be the proven, working path.** New paths start as opt-in; once proven, they become the default with the old path as opt-out.

### 3. Disambiguation Is Bilateral

Telling asset A "you handle this question" without telling asset B "you don't handle this question" leaves both competing. This applies to Genie routing (MV vs TVF) and more broadly to any system where an LLM must choose between alternatives. **Metadata optimization for routing must address BOTH the preferred and competing assets.**
